F-007 & F-015 — Trampa de teclado en el selector de sector

El dropdown "Elige el sector" expone sus 1.540 opciones como botones tabulables en el DOM. Al pulsar Tab, el foco queda atrapado indefinidamente dentro de la lista. Pulsar Escape tampoco libera el foco. Un usuario de teclado o lector de pantalla no puede completar el registro.

Problema actual

Los ítems del listbox son botones con tabindex positivo. Tab los recorre uno a uno sin salida posible.

Abanicos
Abastecedores de buques
Abastecimiento de agua
Abogados…
Tab queda atrapado aquí — 1.540 ítems por recorrer
Lector de pantalla anuncia button, Abanicos
↩ Tab
button, Abastecedores de buques
↩ Escape (sin efecto) ↩ Tab
button, Abastecimiento de agua [+1.537 más…]
Solución propuesta

Patrón ARIA Combobox: el listbox se navega con ↑↓, Tab sale del widget.

Abanicos
Abastecedores de buques
Abastecimiento de agua

↑↓ navegan · Escape cierra · Tab → siguiente campo

Lector de pantalla anuncia textbox, Sector, expanded, required
↓ ArrowDown
Abanicos, option 1 of 3
↓ Escape
textbox, Sector, collapsed
↓ Tab
textbox, Página web (opcional)

Cambios de código necesarios

  • Añadir role="combobox", aria-expanded, aria-haspopup="listbox" al wrapper
  • Los ítems del listbox deben tener role="option" y tabindex="-1" (nunca positivo)
  • Usar aria-activedescendant en el input para indicar el ítem activo
  • Teclado: ↓↑ navegan, Enter selecciona, Escape cierra, Tab sale
<!-- ✗ Antes -->
<button tabindex="0">Abanicos</button>  <!-- ×1540 -->

<!-- ✓ Después -->
<div role="combobox" aria-expanded="true" aria-haspopup="listbox">
  <input aria-autocomplete="list" aria-activedescendant="opt1" />
  <ul role="listbox">
    <li role="option" tabindex="-1">Abanicos</li>
  </ul>
</div>

F-011 — Errores de validación mostrados en la carga inicial

Los campos "Nombre de empresa" y "Sector" se anuncian como invalid con mensajes de error visibles antes de cualquier interacción. WCAG exige que los errores aparezcan solo tras interacción del usuario.

Problema actual

Al cargar la página, los campos ya tienen aria-invalid="true" y alertas activas.

Introduce el nombre de la empresa
Al cargar la página textbox, Nombre de la empresa, invalid, required
alert: Introduce el nombre de la empresa
Solución propuesta

Estado neutral al cargar. El error aparece solo al salir del campo vacío o al intentar enviar.

Sin errores en carga inicial


Introduce el nombre de la empresa
Al cargar la página textbox, Nombre de la empresa, not invalid, required
— sin alertas —

Cambios de código necesarios

  • Inicializar todos los campos con aria-invalid="false" (o sin el atributo)
  • Activar la validación solo en el evento blur (campo vacío) o en submit
  • El contenedor role="alert" debe estar vacío al cargar; añadir el texto dinámicamente
// ✗ Antes — validación al montar
useEffect(() => { validateAll(); }, []);

// ✓ Después — validación tras interacción
<input onBlur={validateField} aria-invalid={hasError} />
<div role="alert">{hasError ? mensajeError : ''}</div>

F-016 — Los mensajes de error no se anuncian al lector de pantalla

Tras salir de un campo vacío (blur), la validación puede mostrar un borde rojo visualmente, pero no actualiza aria-invalid ni inyecta texto en un live region. El lector de pantalla nunca anuncia el error. Además, no existe ningún contenedor role="alert" en el árbol de accesibilidad entre el input y el siguiente campo.

Problema actual

La validación solo añade una clase CSS para el borde rojo. aria-invalid nunca cambia y no existe ningún role="alert".

← visible en pantalla, pero aria-invalid sigue siendo false

Log del lector de pantalla tras blur textbox, Nombre de la empresa, not invalid ← nunca cambia
— sin alerta, sin mensaje de error —
textbox, Elige el sector… not invalid
Solución propuesta

Cada input tiene su role="alert" vacío al cargar. El blur actualiza aria-invalid e inyecta el mensaje.

Pulsa Tab sin escribir nada para ver el error →

Log del lector de pantalla tras blur esperando interacción…

Cambios de código necesarios

  • Añadir aria-describedby="[id-error]" al input y un <span role="alert"> vacío asociado
  • En el handler de blur: setear aria-invalid="true" e inyectar el texto del error en el span
  • En el handler de input: resetear aria-invalid="false" y vaciar el span para limpiar la alerta
<!-- ✗ Antes: solo CSS, sin semántica -->
<input class="input--error" aria-invalid="false" />
<p class="error-msg">Introduce el nombre</p>  <!-- invisible para SR -->

<!-- ✓ Después: semántica + live region -->
<input
  id="nombre"
  aria-invalid="false"
  aria-describedby="nombre-err"
/>
<span id="nombre-err" role="alert" aria-live="assertive"></span>

// JS — validación en blur
input.addEventListener('blur', () => {
  if (!input.value.trim()) {
    input.setAttribute('aria-invalid', 'true');
    errorSpan.textContent = 'Introduce el nombre de la empresa';
  }
});
input.addEventListener('input', () => {
  if (input.value.trim()) {
    input.setAttribute('aria-invalid', 'false');
    errorSpan.textContent = '';
  }
});

F-003 — Orden de encabezados roto por el modal de cookies

El primer encabezado del DOM es un <h2> del modal de cookies ("Valoramos tu privacidad"), antes del <h1> de la página. Los usuarios que navegan con la tecla H encuentran un h2 sin contexto jerárquico.

Estructura DOM actual
<body>
<div class="cookie-modal">
⚠ <h2>Valoramos tu privacidad</h2>
</div>
<header> … </header>
<main>
<h1>Tu empresa en la Guía Útil</h1>
<h2>Háblanos de tu empresa</h2>
</main>
Navegación con tecla H 1· h2 "Valoramos tu privacidad" ← ¿qué h1 lo precede?
2· h1 "Tu empresa en la Guía Útil"
3· h2 "Háblanos de tu empresa"
Estructura DOM corregida
<body>
<header> … </header>
<main>
<h1>Tu empresa en la Guía Útil</h1>
<h2>Háblanos de tu empresa</h2>
</main>
✓ <div role="dialog" aria-modal="true"
aria-labelledby="cookie-title">
<h2 id="cookie-title">Valoramos tu privacidad</h2>
</div>
Navegación con tecla H 1· h1 "Tu empresa en la Guía Útil"
2· h2 "Háblanos de tu empresa"
(modal al final del DOM — no interfiere)

Cambios de código necesarios

  • Mover el modal de cookies al final del <body>
  • Añadir role="dialog", aria-modal="true" y aria-labelledby
  • Al abrir el modal, mover el foco a su interior; al cerrarlo, devolverlo al elemento que lo activó
<div role="dialog" aria-modal="true"
     aria-labelledby="cookie-title">
  <h2 id="cookie-title">Valoramos tu privacidad</h2>
</div>

F-013 — Logo del header sin texto alternativo

El banner aparece vacío en el árbol de accesibilidad. Si el logotipo es el único elemento que identifica la marca en el header, su texto alternativo es obligatorio.

Problema actual
Q
alt ausente

El enlace no tiene nombre accesible

SR sobre el logo link ← sin descripción
Solución propuesta
Q
aria-label="QDQ - Ir a inicio"

Comunica marca y destino

SR sobre el logo link, QDQ - Ir a inicio

Cambios de código necesarios

  • Si el logo es <img>: añadir alt="QDQ"
  • Si es SVG inline: añadir <title>QDQ</title> dentro del SVG
  • Si está dentro de <a>: añadir aria-label="QDQ - Ir a inicio" al enlace e alt="" a la imagen
<!-- ✗ Antes -->
<a href="/"><img src="logo.svg" /></a>

<!-- ✓ Después -->
<a href="/" aria-label="QDQ - Ir a inicio">
  <img src="logo.svg" alt="" />
</a>

F-014 — Indicador de progreso sin nombres descriptivos

Los pasos del registro anuncian "part 1", "part 2", "part 3" en lugar de sus nombres reales. Un usuario de lector de pantalla no sabe qué implica cada etapa del proceso.

Problema actual

Paso 1 de 3

part 1
part 2
part 3
SR anuncia Paso 1 de 3
part 1 · part 2 · part 3
Solución propuesta
  1. 1. Datos de empresa (actual)
  2. 2. Datos de contacto
  3. 3. Confirmación
SR anuncia navigation, Progreso del registro
1. Datos de empresa (actual)
2. Datos de contacto
3. Confirmación

Cambios de código necesarios

  • Envolver en <nav aria-label="Progreso del registro">
  • Usar <ol> + <li> con el nombre real de cada paso
  • Marcar el paso activo con aria-current="step"
<nav aria-label="Progreso del registro">
  <ol>
    <li aria-current="step">Datos de empresa</li>
    <li>Datos de contacto</li>
    <li>Confirmación</li>
  </ol>
</nav>
